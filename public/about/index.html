关于我

背景经历，以及我如何看待系统与架构

关于我#

我是一名 企业级平台架构负责人， 拥有 10 年 Java 经验，其中 9 年从事架构相关工作，角色经历包括：

  • 架构成员
  • 架构师
  • 架构负责人

长期负责 平台级系统的设计、演进与技术治理。

关键经历#

企业级中台 0 → 1（2017）#

2017 年，我作为小组长参与并主导了 企业级业务中台的早期建设：

  • 从 0 到 1 搭建中台服务

  • 所在小组的服务是 首批上线的中台服务

  • 深度参与：

    • 服务边界划分
    • 能力抽象与下沉
    • 中台与业务系统的职责划分

这一阶段对我理解 “平台应承担什么，不应承担什么” 产生了决定性影响。

架构演进阶段#

随着系统规模扩大，我的角色逐步演进为架构负责人，主要工作包括：

  • 平台级架构设计与技术路线规划
  • 多业务线共用平台的演进治理
  • 服务解耦与能力沉淀
  • 系统级技术债的识别与治理

重点不在“重构本身”， 而在 让系统在组织变化中仍然可控。

当前关注方向#

目前我主要负责 整体技术架构与平台演进，并推动：

  • AI 能力与传统中台体系的融合
  • 基于 海量数据 的 RAG 架构
  • 多上下文检索、用户向量、对话历史管理
  • 面向业务的智能体能力封装

我如何看待系统与架构#

我并不追求：

  • 短期交付速度
  • 炫技式架构设计
  • 脱离组织现实的“理想方案”

我更关注：

  • 系统在 3～5 年后是否仍可演进
  • 复杂度是被 收敛 还是在 扩散
  • 架构决策是否 长期可解释、可维护

好的架构不是聪明， 而是让系统在不断变化中依然可控。

关于这个站点#

这是一个 长期维护的个人技术站点，主要记录：

  • 架构设计与演进思考
  • 平台化、中台化的实践经验
  • AI 系统在真实业务中的落地方式

它不是教程合集， 而是 在真实系统中反复验证后的沉淀记录。

一些非工作的事情#

工作之外，我喜欢 跑步。

不是为了成绩或装备，更多是当作一种 让思路变得干净、从复杂问题中抽离出来的方式。

很多系统层面的想法，反而是在跑步的时候想清楚的。
